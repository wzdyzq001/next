# 团小帮 - 生活服务订单 AI 智能履约助手 V2.5 - Product Requirement Document

## Overview
- **Summary**: 基于现有 V2.1 代码增量迭代升级"团小帮"AI 智能履约助手至 v2.5，构建全栈 Demo。前端在现有 React 18 + TypeScript + Vite + 纯 CSS 代码基础上扩展功能组件和流程，后端新建 Node.js + Fastify + TypeScript + SQLite + Prisma 独立服务提供 REST API + WebSocket。覆盖三大入口、22 种消息触达类型、四层订单卡片架构、7+1 类功能卡片、退款挽留机制、待支付专项设计、引导问题智能排序、多轮对话上下文、WebSocket 实时状态同步、AI 三级降级、兜底转人工/商家、商家端联动（紧急通知+预设问题+状态同步+权限管控）等全量能力。
- **Purpose**: 通过 AI 智能助手为用户提供订单全生命周期的自然语言交互能力，覆盖订单查询、履约进度追踪、功能操作等场景，结合气泡推送主动触达，提升用户履约体验与问题解决效率，降低退款率、提升支付转化率。
- **Target Users**: 抖音生活服务订单用户（餐饮、酒店、景区、综合、旅行社五大行业）+ Demo 演示场景。

## Goals
- 基于 V2.1 已有代码增量升级前端 AI 助手组件（复用现有 Provider/浮层/气泡/订单卡片/对话引擎基础架构），补充 v2.5 新增能力
- 实现三种入口：列表页右上角（固定入口A）、详情页左下角（固定入口B，待支付除外）、气泡推送（动态入口）
- 实现 AI 助手浮层（9 分屏默认 / 全屏上滑 / 关闭三态平滑过渡）
- 实现四层订单卡片架构（基础层+扩展层+操作层+引导问题层），覆盖全行业×商品类型×订单状态组合矩阵
- 实现扩展层智能裁剪（首次完整/追问精简/气泡突出/多卡隐藏/退款突出）
- 实现 22 种消息触达类型（§3.3），含优先级排序、多消息轮转（最多3条）、极端天气正向牵引、游玩攻略 AI 生成内容
- 实现三种气泡形式（常驻短/临时短/长气泡→自动收折）及频控策略
- 实现 7 类功能卡片：使用提醒（快捷选项+自定义天数+双向联动+去重规则）、预约表单、加急请求（15min冷却）、再来一单、退款申请、攻略/指引、商家紧急通知
- 实现退款挽留机制（自然语言触发→稀缺性挽留→坚持退款→三因子退款能力判断→退款卡片）
- 实现待支付专项设计（无左下角入口、无气泡、支付倒计时、取消挽留稀缺性四维度）
- 实现引导问题层（多源数据驱动，最多3个，三层排序模型 P0×0.5+F×0.3+C×0.2，排除清单，对话轮次联动）
- 实现多轮对话上下文管理（30min会话过期、消息20条上限、跨订单清理、待操作15min过期）
- 实现 WebSocket 实时状态同步 + 15s 轮询兜底，5 种卡片状态更新规则
- 实现 AI 三级降级（L1部分降级/L2完全降级/L3引导问题降级）+ 灰度恢复
- 实现 AI 能力边界与兜底机制（转人工客服5触发条件、转商家沟通3触发条件、8类无法解决场景）
- 实现用户侧商家联动展示：紧急通知卡片（mock触发）、预设问题带「商家」标签展示（排序加权δ=0.15）、履约状态同步（mock触发）
- 商家端工作台/发送界面不在本期范围，后续迭代
- 实现数据埋点钩子（前端预留，Demo 中控制台打印）
- 搭建独立后端服务，提供核心 REST API + WebSocket
- Demo 含高保真交互，可走完全部核心场景链路

## Non-Goals (Out of Scope)
- 真实 AI/NLP 大模型接入（对话响应通过规则引擎 + Mock 话术模拟，引导问题使用预置静态数据矩阵）
- 语音输入功能（仅占位 UI）
- 支付真实对接（mock 支付流程）
- 真实导航/电话/地图跳转（toast/alert 模拟）
- 生产环境部署与监控告警体系（降级监控用控制台日志）
- CPO/CCR/IM/电话真实数据管线（引导问题用预置静态数据 + mock 排序分数）
- 内容安全审核机审/人审
- 商家端工作台页面（商家发送通知/管理预设问题界面后续迭代）
- 多语言支持、无障碍设计、离线场景（PRD §25 待补充项）
- A/B 实验框架

## Background & Context
- 项目技术栈：前端 React 18 + TypeScript + Vite + 纯 CSS（CSS Modules 或 BEM 命名空间），不引入 UI 组件库
- 后端技术栈：Node.js + Fastify + TypeScript + SQLite + Prisma ORM，独立目录管理
- 现有 V2.1 代码已有基础框架（Provider、浮层、气泡等），但用户要求完全重写，不直接复用代码，但可参考架构思路
- 品牌名：团小帮；Slogan：全程陪伴，省心又省力
- 入口复用原则：团小帮复用已有 AI 智能助手入口位置，不新建独立入口；代码全部重新实现
- 移动端适配基准：375px 视口，支持 320-430px 宽度

## Functional Requirements

### FR-1: 入口矩阵（§3）
- **FR-1.1**: 订单列表页右上角固定入口A：AI 图标 + 未读红点（8×8px+白色描边），点击打开 9 分屏浮层，不默认带入订单卡片。无历史记录展示通用欢迎语+引导问题；有历史记录折叠展示最近3-4条。
- **FR-1.2**: 订单详情页左下角固定入口B：非待支付订单展示圆形 AI 图标，点击打开浮层并带入当前订单卡片（锁定上下文）；待支付订单此处展示应付金额（红色大字），不展示 AI 入口。同订单重复进入不重复带入卡片。
- **FR-1.3**: 动态气泡入口：系统基于订单状态机主动推送，独立于入口图标。点击气泡进入助手，气泡内容作为上下文带入，自动围绕气泡内容展开。
- **FR-1.4**: 历史记录折叠策略：列表入口默认折叠最近3-4条；聊天过程中不折叠；详情页入口重复进入不折叠。

### FR-2: 消息触达矩阵（§3.3）
- **FR-2.1**: 实现 22 种消息类型的完整覆盖矩阵（极端天气橙/红/黄、演出变更、退款提醒、门店客流/库存、商品库存、闭店提醒、临期提醒、入住提醒、出行提醒、商品变更、出行商品变更、商家扫码、景区提示、防诈提示、游玩须知、自动核销提醒、全部门店停业、部分门店停业、订单延期、消费券限制、先用后付扣款失败、游玩攻略）
- **FR-2.2**: 按行业分类的消息覆盖矩阵（餐饮12种、综合+演出、游玩11种、住宿6种、旅行社6种）
- **FR-2.3**: 消息优先级排序（0最高→最低），单次进入详情页信息通知条最多轮转3条，按优先级降序
- **FR-2.4**: 极端天气专项：18种气象类型、6大天气分类、正向牵引策略（不引导改期/退款，按天气类型提供实用出行建议）
- **FR-2.5**: 游玩行业专项：游玩须知（证件/换票/预约/入园条件）+ 游玩攻略（推荐入园时段/时长/里程/摘要/路线规划）
- **FR-2.6**: 消息承接逻辑：有消息时点击消息条→带入消息内容+订单卡片；未点击直接进AI→主动展示消息摘要+"您有N条订单提醒待查收"
- **FR-2.7**: 传递给AI助手的消息参数：order_id、order_status、validity_period、product_id、nearest_store_poiID、reservation_store_poiID、is_same_city、is_at_store、message_list、weather_warning、play_guide

### FR-3: 气泡推送能力（§4）
- **FR-3.1**: 常驻短气泡（≤7字）：履约异常高优先级提醒，常驻展示可点击
- **FR-3.2**: 临时短气泡（≤7字）：弹出后3-5秒自动隐藏，展示期间不可直接交互
- **FR-3.3**: 长气泡→自动收折：展开态≤30字/1行，5s或滚动后收折为≤7-8字短气泡，收折后可点击，不再自动展开
- **FR-3.4**: 气泡推送场景规则：只在有价值节点推送，常规状态变更（支付成功、商家接单等）不重复推送
- **FR-3.5**: 气泡频控：同一订单每个状态节点最多1次；待支付不推送；用户可配置关闭
- **FR-3.6**: 气泡信息带入：解析order_id+event_type→加载订单卡片→基于event_type生成上下文引导（配送延迟/退款审核/闭店提醒/临期提醒四种示例场景）

### FR-4: AI助手浮层界面（§5）
- **FR-4.1**: 浮层三态：9分屏（默认，top:10%，顶部露出底层页面）→全屏（top:0，上滑触发）→关闭（下滑/点击外部/关闭按钮）
- **FR-4.2**: 浮层交互：默认9分屏打开、上滑拉全屏、全屏下滑恢复9分屏、快捷关闭（点击顶部露出区域）、状态保持（关闭后再打开恢复上下文）、底层页面冻结
- **FR-4.3**: 顶部区域：AI Icon + "团小帮" + "全程陪伴，省心又省力" + 关闭按钮
- **FR-4.4**: 中间聊天区域：支持纯文本/订单卡片/功能卡片/引导问题/富信息卡片（配送地图、进度时间轴、取餐码）
- **FR-4.5**: 多轮对话：上下文记忆、卡片可交互、引导问题动态生成（最多3个）、消息带入展示
- **FR-4.6**: 底部输入区：订单选择器（点击展开订单列表，选中后上方展示tag）、语音输入占位、文本输入框、发送按钮

### FR-5: 订单卡片带入场景与上下文（§6）
- **FR-5.1**: 四种带入场景：详情页入口（锁定订单）、手动选择（切换订单）、自然语言查询（AI解析匹配）、气泡入口（锁定订单+事件）
- **FR-5.2**: 上下文管理规则：单订单锁定、多订单AI展示多卡片、切换订单重置、跨订单对比汇总、气泡带入锁定事件

### FR-6: 四层订单卡片架构（§7）
- **FR-6.1**: 基础层（必选）：缩略图+商品名+价格+标签+距离+门店名+[导航][电话]+状态角标
- **FR-6.2**: 扩展层（可选）：按行业×状态动态展示（履约进度/入住信息/退款信息/取餐码/支付倒计时/配送地图等）
- **FR-6.3**: 操作层（可选）：按状态机展示按钮，双模式（功能卡片/直接跳转），退款按钮不外露
- **FR-6.4**: 引导问题层（AI独有）：最多3个预测问题，基于履约节点+多源反馈+对话上下文动态生成
- **FR-6.5**: 多维度组合矩阵：行业(Category)×商品类型(ProductType)×履约方式(RedeemMethod)×订单状态(Status)
- **FR-6.6**: 扩展层AI智能裁剪：首次完整、追问精简、气泡突出、多卡隐藏、退款突出
- **FR-6.7**: 多卡片展示规则：查询最近订单→纵向精简列表；快到期→列表+汇总标注；多商品分批→主卡完整+子卡精简；跨订单对比→精简+对比标注

### FR-7: 操作层全状态规则（§8）
- **FR-7.1**: 操作按钮双模式：功能卡片模式（聊天内展开需确认的操作）vs 直接跳转模式（简单查看类操作）
- **FR-7.2**: 7类功能卡片触发规则：使用提醒、预约表单、加急请求、再来一单、退款申请（自然语言触发）、一站式游玩攻略（仅景区）、入住指引（仅酒店）、出行指引（仅旅行社）
- **FR-7.3**: 行业×商品类型订单状态机：完整覆盖餐饮（6种核销组合+点单/配送全流程）、酒店（预售券+日历房）、景区（团购+预售券+日历票）、综合（团购+演出+KTV+电影）、旅行社（预售券含待出行/行程中）
- **FR-7.4**: 餐饮6种核销方式组合按钮动态展示：纯券码/纯点单/点单+券码/配送+券码/点单+配送/点单+配送+券码
- **FR-7.5**: 餐饮"提前预约免排队"仅正餐二级类目展示
- **FR-7.6**: 旅行社待出行（出行日前倒计时+出行指引）、行程中（标签展示无操作按钮）
- **FR-7.7**: 日历房/日历票交易完成/取消/退款成功增加[再次预订]按钮
- **FR-7.8**: 申请退款不在操作层外显，仅自然语言触发+AI挽留后处理

### FR-8: 退款能力与挽留机制（§9）
- **FR-8.1**: 退款入口仅自然语言通道：操作层/引导层均不展示退款按钮/问题
- **FR-8.2**: 退款能力三因子判断模型：f(当前时间, 商家退款规则, 用户权益) → full_refund/partial_refund/non_refundable/cancelable
- **FR-8.3**: 挽留机制：用户表达退款意图→AI挽留干预（优惠稀缺+库存稀缺+使用价值+退款损耗+替代方案）→用户决策（继续使用/仍要退款）
- **FR-8.4**: 各场景挽留策略差异化：团购券待使用、预售券待预约、预售券已预约、日历房/票已确认、配送中、点单中
- **FR-8.5**: 挽留后用户决策处理：继续使用→触发对应流程；再次明确退款→不再挽留触发退款流程；问其他问题→正常响应
- **FR-8.6**: 退款功能卡片：退款金额+后端动态返回退款说明文案+原因四选一（不想要了/商家原因/计划有变/其他）+[取消][确认退款]
- **FR-8.7**: 退款文案动态展示：标签(productRules.refundRule)+功能卡片(后端实时返回)+AI回复(后端实时返回)

### FR-9: 待支付状态专项（§10）
- **FR-9.1**: 待支付订单详情页左下角不展示AI入口（展示应付金额），不推送气泡
- **FR-9.2**: 待支付仅通过列表页右上角入口进入
- **FR-9.3**: 待支付卡片：基础层+扩展层（支付倒计时内联，左倒计时+右红色立即支付）+操作层仅[立即支付]（红色强调）
- **FR-9.4**: 待支付引导问题：P0消除顾虑（随时退吗？）+P1商品信息+P2门店信息，各行业差异化
- **FR-9.5**: 待支付取消挽留：四维度稀缺性（优惠+库存+活动+历史），文案组合规则（1维度直展/2维度用"且"连接/3+维度取TOP2）
- **FR-9.6**: 稀缺性信息基于后端真实数据，AI不编造
- **FR-9.7**: 倒计时到期处理：≤5分钟变红闪烁+临时气泡"即将到期"；到期→已取消+[重新下单]

### FR-10: 引导问题层设计（§11）
- **FR-10.1**: 多源数据驱动预测式推荐：CPO平台+CPO数据集+CCR数据集+在线IM+电话咨询5类数据源
- **FR-10.2**: 数据管线架构：多源采集→ETL(清洗/归类/聚合/排序/过滤/话术优化)→预测问题候选池→展示层
- **FR-10.3**: 引导问题生成规则：P0(反馈量TOP1-2)+P1(扩展层延伸)+P2(操作层引导)+P3(上下文追问)
- **FR-10.4**: 全行业全状态预测问题矩阵：覆盖餐饮/酒店/景区/综合/旅行社各履约节点
- **FR-10.5**: 展示规则：最多3个、降序排列、去重、动态更新、行业感知、正向牵引、排除清单
- **FR-10.6**: 排除清单：未描述明确诉求、功能异常、投诉商家、食品安全、异物、虚假宣传、投诉客服、钱财被骗、受伤等负向/严重问题不主动引导
- **FR-10.7**: 数据更新策略：每日T-1同步、滚动7天日均、每周全量刷新权重、异常监控、灰度发布、数据源降级

### FR-11: 通用组件（§12）
- **FR-11.1**: FulfillmentTimeline：N节点N-1段，done(实心✓)/current(脉冲动画)/pending(空心)，进度百分比=已完成段数/总段数
- **FR-11.2**: CollapsibleTimeline：默认折叠，点击展开/收起，折叠行背景#F3F4F6圆角8px，用于已取餐/已送达
- **FR-11.3**: RefundingInfo：单行左状态文案+右退款金额，无进度模块
- **FR-11.4**: RefundSuccessInfo：单行左"已原路退回"+右金额，灰色文字弱化
- **FR-11.5**: PickupCodeDisplay：有码→二维码+大字取餐码；无码→"查看取餐码"按钮
- **FR-11.6**: 支付倒计时内联：左倒计时+右红色立即支付，嵌入扩展层无额外底框
- **FR-11.7**: 配送地图卡片（AI独有）：骑手实时位置+距离标注+实时刷新

### FR-12: 功能卡片完整设计（§17）

### FR-13: 引导问题排序算法（§18）
- **FR-13.1**: 三层排序模型：离线权重层P0(α=0.5)+在线反馈学习层F(β=0.3)+上下文调整层C(γ=0.2)
- **FR-13.2**: P0离线权重：滚动7天日均反馈量归一化(P0_score=该问题/该节点TOP1)，按行业×商户×商品×状态四维聚合
- **FR-13.3**: F在线反馈：14天滚动CTR归一化，冷启动0.5，线性衰减，展示量<100不纳入，CTR突降>50%告警
- **FR-13.4**: C上下文调整：已回答→0、关键词匹配→+0.3、扩展层关联→+0.2、操作层关联→+0.1、P1/P2/P3规则叠加
- **FR-13.5**: 综合得分公式：FinalScore = 0.5×P0 + 0.3×F + 0.2×C
- **FR-13.6**: 排序流程：查候选池TOP10→算P0→算F→算C→综合得分→降序取TOP3→埋点上报→CTR回流
- **FR-13.7**: 特殊规则：新问题灰度10%、排除清单P0=0、退款问题仅自然语言触发后展示、跨行业通用问题共享CTR、多卡片场景精简卡不展示引导问题
