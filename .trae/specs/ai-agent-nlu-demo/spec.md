# AI 助手自然语言交互 Demo - Product Requirement Document

## Overview
- **Summary**: 在 AI 助手 Demo 中模拟基于自然语言的四大核心功能交互：帮我约、订单使用提醒、取餐码查询、配送进度查询。通过多轮对话引导用户完成信息收集，最终生成对应的功能卡片（预约卡片、提醒卡片、取餐码展示、配送进度展示）。
- **Purpose**: 演示 AI 助手如何通过自然语言理解用户意图，通过多轮对话补全必要信息，并最终完成服务预约/查询的完整流程。
- **Target Users**: 产品经理、设计师、开发人员，用于需求评审、交互演示和开发参考。

## Goals
- 实现"帮我约"功能的自然语言交互，支持多轮对话收集日期、时间、人数信息
- 实现"订单使用提醒"功能的自然语言交互，支持临期提醒气泡和提醒设置
- 实现"取餐码查询"功能的自然语言交互，支持有无前置订单卡片两种场景
- 实现"配送进度查询"功能的自然语言交互，参考取餐码查询逻辑
- 所有功能均需在 Demo 中可交互演示，状态流转完整

## Non-Goals (Out of Scope)
- 真实的后端 API 对接（仅前端 Mock 数据）
- 真实的 NLP/ASR 语义理解（仅关键词匹配模拟）
- 真实的预约/提醒服务对接（仅本地状态管理）
- 多轮对话的上下文记忆扩展（仅当前会话有效）
- 移动端适配优化（保持现有布局即可）

## Background & Context
- 现有项目已实现 AI 助手基础框架，包含消息列表、订单卡片、预约面板、提醒设置等基础组件
- 已有帮我约功能的基础实现（预约面板、预约卡片、状态管理）
- 已有订单使用提醒的基础实现（提醒设置、提醒状态管理）
- 已有订单卡片组件（FullOrderCard、CompactOrderCard）
- 需要在现有基础上，增加自然语言交互的模拟层，通过关键词识别意图，通过多轮对话收集信息

## Functional Requirements

### FR-1: 帮我约 - 自然语言交互
- 展示订单卡片时，根据订单类型和时间场景显示额外预约提示（如"门店客流量大，提前预约免排队"、"临近假期门店繁忙，建议提前预约"等）
- 识别用户输入中的预约意图关键词（预约、帮我约、立即预约、马上预约等）
- 识别到预约意图后，回复引导语并提供"帮我约"快捷按钮
- 多轮对话收集三项信息：预约日期、预约时间、就餐人数
- 支持自然语言日期解析（明天、后天、这周几、下周几等）
- 支持自然语言人数解析（数字、汉字数字等）
- 缺失信息时主动追问
- 日期/时间不可预约时，提示可约时间供用户选择
- 信息收集完成后自动发起预约，生成预约卡片

### FR-2: 订单使用提醒 - 自然语言交互
- 订单临期时，AI 助手卡片气泡提醒（长文案几秒后收起为短文案"临期提醒"）
- 用户进入 AI 助手时，如订单临期，额外增加提醒引导文案
- 识别用户输入中的提醒意图关键词（提醒、设置、帮我提醒、提醒我等）
- 识别到提醒意图后，回复引导语并展示快捷选项组件
  - 快捷选项包括：明天、后天、本周五/六/日、下周五/六/日、最后一天、过期前1天、过期前3天
  - 日期重叠时优先展示近日期表达（优先级：明天/后天 > 本周X > 下周X）
  - 所有可选日期不超过订单有效期，超出的选项自动过滤不展示
- 支持自然语言日期解析（明天、后天、这周几、下周几、过期前几天等）
- 校验提醒日期是否超过订单有效期，超期时提示并建议过期前一天提醒
- 用户确认后自动生成使用提醒卡片

### FR-3: 取餐码查询 - 自然语言交互
- 识别用户输入中的取餐码查询意图（取餐码、取餐、取号等）
- **无前置订单卡片场景**：
  - 查询用户待取餐订单
  - 找到1个：发送订单卡片 + 取餐码展示
  - 找到多个：展示多个简化订单卡片（基础信息+取餐码）
  - 未找到：回复提示 + "选择订单"按钮
- **有前置订单卡片场景**：
  - 非餐饮订单：提示无取餐码 + "选择订单"按钮
  - 餐饮订单：
    - 取消/退款状态：提示无取餐码 + "选择订单"按钮
    - 待使用状态：
      - 仅券码核销：提示无取餐码 + "查看券码"按钮
      - 支持点单核销：提示无取餐码 + "是否点单"是/否选项
      - 仅配送：提示无取餐码 + "是否预约配送"是/否选项
    - 交易完成状态：
      - 券码核销（抖音端内）：提示无取餐码 + "选择订单"按钮
      - 券码核销（端外，商家App/微信小程序）：提示"没有找到取餐码，如果在商家App/微信小程序核销，可通过原渠道查找取餐码"
      - 点单核销：展示取餐码 + 取餐进度
      - 配送核销：提示配送订单 + 展示配送进度

### FR-4: 配送进度查询 - 自然语言交互
- 识别用户输入中的配送查询意图（配送、物流、送到哪了、骑手等）
- 参考取餐码查询逻辑，区分有无前置订单卡片两种场景
- 有配送中的订单时，展示配送进度条、骑手信息、预计送达时间
- 无配送订单时，引导用户选择订单或提示无配送订单

### FR-5: 交互体验优化
- 快捷选项按钮（快速回复）支持点击即发送，使用现有 quickReplies 样式
- 多轮对话状态保持，用户发送无关消息时按新消息响应（不中断原流程，新消息按新意图处理）
- 状态流转有视觉反馈（卡片更新、消息动画等）
- 支持用户随时取消当前流程
- 取餐码/配送查询场景中，支持模拟点单核销状态切换（待使用 → 已确认 → 备餐中 → 可取餐 / 配送中）

## Non-Functional Requirements
- **NFR-1**: 交互响应延迟 < 500ms（模拟 AI 思考时间 300-800ms）
- **NFR-2**: 关键词识别准确率 ≥ 90%（针对 Demo 中预设的关键词）
- **NFR-3**: 所有功能均可在 Demo 中完整走通，无死胡同
- **NFR-4**: 代码结构清晰，便于后续接入真实 NLP 服务

## Constraints
- **Technical**: React 18 + TypeScript + Vite，使用现有 AiAssistantProvider 状态管理
- **Business**: 纯前端 Mock，不依赖后端服务
- **Dependencies**: 复用现有组件（FullOrderCard、ReservationPanel、提醒设置组件等）

## Assumptions
- 用户输入通过关键词匹配识别意图，不需要真实的 NLP
- 日期解析仅支持常见自然语言表达（明天、后天、周X、下周X、过期前X天等）
- 所有数据均为 Mock 数据，不需要真实接口调用
- 多轮对话状态仅在当前会话有效，刷新后重置

## Acceptance Criteria

### AC-1: 帮我约 - 预约提示展示
- **Given**: 用户打开 AI 助手并带入一个需要预约的订单（综合类或餐饮正餐类）
- **When**: 订单卡片展示时
- **Then**: 卡片下方或消息中显示额外的预约引导提示（如"门店客流量大，提前预约免排队"），提示语根据场景动态变化
- **Verification**: `human-judgment`
- **Notes**: 至少有 3 种不同的提示语模板

### AC-2: 帮我约 - 意图识别与引导
- **Given**: 用户在 AI 助手中输入包含"预约"、"帮我约"等关键词的消息
- **When**: AI 助手识别到预约意图
- **Then**: 回复"请告诉我你要预约什么时间、几个人，我会帮你预约最近的门店"，并附带"帮我约"快捷按钮
- **Verification**: `human-judgment`

### AC-3: 帮我约 - 多轮信息收集
- **Given**: 用户已触发预约流程
- **When**: 用户只提供了部分信息（如只说了日期）
- **Then**: AI 助手追问缺失的信息（时间、人数）
- **Verification**: `human-judgment`
- **Notes**: 需要覆盖：只有日期、只有时间、只有人数、日期+时间缺人数等各种缺失组合

### AC-4: 帮我约 - 日期时间校验
- **Given**: 用户提供了预约日期和时间
- **When**: 该时间不可预约（已过期或门店休息）
- **Then**: AI 助手提示该时间不可约，并展示可约的日期/时间供用户选择
- **Verification**: `human-judgment`

### AC-5: 帮我约 - 生成预约卡片
- **Given**: 用户已提供完整的预约信息（日期、时间、人数）
- **When**: 信息校验通过
- **Then**: 自动发起预约，AI 助手发送预约卡片，状态为"预约确认中"
- **Verification**: `human-judgment`

### AC-6: 订单使用提醒 - 临期气泡提醒
- **Given**: 订单临近有效期（如剩余 ≤ 7 天）
- **When**: 用户打开订单详情页或 AI 助手，订单卡片展示后延迟 1.5 秒
- **Then**: AI 助手卡片旁弹出气泡提醒"订单还有 X 天过期，请尽快使用～"，5 秒后气泡收起为短消息"临期提醒"
- **Verification**: `human-judgment`

### AC-7: 订单使用提醒 - 意图识别与快捷选项
- **Given**: 用户输入包含"提醒"、"设置"等关键词
- **When**: AI 助手识别到提醒设置意图
- **Then**: 回复"请告诉我提醒时间"，并展示快捷选项组件
  - 快捷选项包括：明天、后天、本周五/六/日、下周五/六/日、最后一天、过期前1天、过期前3天
  - 日期重叠时优先展示近日期（如明天=本周五时，显示"明天"而非"本周五"）
  - 所有可选日期均不超过订单有效期，超出的选项不展示
- **Verification**: `human-judgment`

### AC-8: 订单使用提醒 - 有效期校验
- **Given**: 用户设置的提醒日期超过了订单有效期
- **When**: AI 助手校验日期后
- **Then**: 提示"提醒日期不可以超过订单有效期哦，是否需要在过期前一天提醒？"，提供是/否选项
- **Verification**: `human-judgment`

### AC-9: 订单使用提醒 - 生成提醒卡片
- **Given**: 用户已选择或输入提醒日期，且日期有效
- **When**: 信息确认后
- **Then**: 自动生成使用提醒卡片，显示提醒日期和订单信息
- **Verification**: `human-judgment`

### AC-10: 取餐码查询 - 无前置订单卡片
- **Given**: AI 助手中没有订单卡片，用户输入"取餐码"
- **When**: AI 助手查询待取餐订单
- **Then**:
  - 找到1个：回复"帮你找到一个待取餐订单" + 订单卡片（含取餐码）
  - 找到多个：展示多个简化订单卡片（基础信息+取餐码）
  - 未找到：回复"没有找到待取餐订单" + "选择订单"按钮
- **Verification**: `human-judgment`

### AC-11: 取餐码查询 - 有前置订单卡片（非餐饮）
- **Given**: AI 助手中有一个非餐饮类订单卡片，用户输入"取餐码"
- **When**: AI 助手判断订单类型
- **Then**: 回复"您咨询的订单不是餐饮订单，没有取餐码，是否要咨询其他订单" + "选择订单"按钮
- **Verification**: `human-judgment`

### AC-12: 取餐码查询 - 有前置订单卡片（餐饮待使用）
- **Given**: AI 助手中有一个餐饮类待使用订单卡片，用户输入"取餐码"
- **When**: AI 助手判断履约模式
- **Then**:
  - 仅券码核销：提示无取餐码 + "查看券码"按钮
  - 支持点单核销：提示无取餐码 + "是否需要我帮你点单？"是/否选项
  - 仅配送：提示无取餐码 + "是否需要我帮你预约配送？"是/否选项
- **Verification**: `human-judgment`

### AC-13: 取餐码查询 - 有前置订单卡片（餐饮已完成）
- **Given**: AI 助手中有一个餐饮类已完成订单卡片，用户输入"取餐码"
- **When**: AI 助手判断实际履约方式
- **Then**:
  - 券码核销（抖音端内）：提示无取餐码 + "选择订单"按钮
  - 券码核销（端外，商家App/微信小程序）：提示"没有找到取餐码，如果在商家App/微信小程序核销，可通过原渠道查找取餐码"
  - 点单核销：展示取餐码 + 取餐进度
  - 配送核销：提示配送订单 + 展示配送进度
- **Verification**: `human-judgment`

### AC-14: 配送进度查询 - 有配送中订单
- **Given**: 用户输入"配送"、"物流"等关键词，且存在配送中订单
- **When**: AI 助手识别到配送查询意图
- **Then**: 展示配送进度（进度条、骑手信息、预计送达时间）
- **Verification**: `human-judgment`

### AC-15: 快捷按钮交互
- **Given**: AI 消息中包含快捷选项按钮
- **When**: 用户点击按钮
- **Then**: 自动发送对应内容，并触发相应流程
- **Verification**: `human-judgment`
