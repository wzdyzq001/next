# 团小帮 - 生活服务订单 AI 智能履约助手 V2.1 - Product Requirement Document

## Overview
- **Summary**: 基于现有 AI 智能助手 V2.0（订单中心）代码，重新实现"团小帮"AI 智能履约助手。助手以浮层形式覆盖在订单列表/详情页上方，提供订单全生命周期的自然语言交互能力，支持三种入口（列表右上角、详情页左下角、气泡推送），四层订单卡片架构（基础层+扩展层+操作层+引导问题层），功能卡片操作，退款挽留机制，待支付专项设计，多轮对话上下文，引导问题智能排序。
- **Purpose**: 提升用户在订单履约全流程中的问题解决效率，通过 AI 主动触达（气泡推送）和自然语言交互降低用户操作成本，通过退款挽留、支付转化等机制提升平台业务指标。
- **Target Users**: 抖音生活服务订单用户（餐饮、酒店、景区、综合、旅行社五大行业）。

## Goals
- 复用已有 AI 助手入口（右上角图标、左下角图标），不新建独立入口
- 实现三种入口形态：固定入口A（列表页右上角）、固定入口B（详情页左下角，待支付除外）、动态气泡推送
- 实现 AI 助手浮层界面（9分屏默认/全屏上滑/关闭三种形态）
- 实现四层订单卡片架构（基础层必选 + 扩展层可选 + 操作层可选 + 引导问题层 AI 独有）
- 实现全行业×商品类型×订单状态的操作层按钮规则
- 实现退款挽留机制（自然语言触发→AI挽留→坚持退款→退款能力判断→退款卡片）
- 实现待支付状态专项设计（无左下角入口、无气泡、支付优先引导、取消挽留）
- 实现引导问题层（多源数据驱动，最多3个，数据排序算法）
- 实现功能卡片（使用提醒设置、预约表单、加急请求、再来一单、退款申请、攻略指引）
- 实现三种气泡形式（常驻短气泡≤7字、临时短气泡≤7字3-5s隐藏、长气泡展开≤30字自动收折）
- 实现多轮对话上下文管理（会话级30min过期、跨订单清理、引导问题联动）
- 实现 WebSocket 实时状态同步 + 15s 轮询兜底
- 实现 AI 降级策略（L1部分降级/L2完全降级/L3引导问题降级）
- 搭建独立后端服务（Node.js + Fastify + TypeScript + SQLite + Prisma），提供核心 REST API + WebSocket
- 前端 Demo 含 Mock 对话能力，可走完核心场景交互流程

## Non-Goals (Out of Scope)
- 商家端联动能力（PRD §24）——紧急通知推送、商家预设常见问题、商家工作台、权限管控等
- 数据埋点上报体系与效果度量后台（PRD §21）——前端预留埋点钩子但不做完整上报
- 真实 AI/NLP 大模型接入——对话响应通过规则引擎 + Mock 话术模拟
- 语音输入功能
- 支付真实对接（使用 mock 支付流程）
- 退款真实流程对接（使用 mock 退款）
- APP 端原生能力（导航、电话、地图跳转用 mock 或页面提示）
- 生产环境部署与监控告警
- 多端适配（仅做移动端 Web 视口）
- CPO/CCR/IM/电话数据管线（引导问题用预置静态数据）

## Background & Context
- 项目技术栈：前端 React 18 + TypeScript + Vite + 纯 CSS + 内联 SVG
- 后端技术栈：Node.js + Fastify + TypeScript + SQLite + Prisma ORM，独立项目分仓管理
- 现有 V2.0 代码已实现：订单列表、订单详情、UnifiedOrderCard 统一卡片、16 种场景 mock 数据、酒店/景区/餐饮/综合等行业状态管理、核销提醒、预约卡片、取餐码展示等
- 入口复用原则：团小帮复用已有 AI 智能助手入口（右上角 AIAssistantIcon），不新建独立入口；开发参考 V1.1 设计风格但代码与逻辑全部重新实现
- V2.1 代码在 `feature/ai-assistant-v2.1` 分支上开发，与 main 分支隔离，方便回退
- 品牌名：团小帮；Slogan：全程陪伴，省心又省力

## Functional Requirements

### FR-1: 入口矩阵
- **FR-1.1**: 订单列表页右上角固定入口A：复用现有 AIAssistantIcon，点击打开 AI 助手浮层（9分屏），不带入订单卡片
- **FR-1.2**: 订单详情页左下角固定入口B：在订单详情页底部左侧展示 AI 助手圆形图标（非待支付订单），点击打开助手并带入当前订单卡片；待支付订单此处展示应付金额而非 AI 入口
- **FR-1.3**: 动态气泡推送：独立于入口图标，由系统基于订单状态+业务事件主动推送三种气泡（常驻短/临时短/长气泡→自动收折），点击气泡进入助手并带入气泡内容+订单上下文

### FR-2: 气泡推送能力
- **FR-2.1**: 常驻短气泡（≤7字）：履约异常提醒（配送延迟、门店闭店等高优先级），常驻展示，可点击
- **FR-2.2**: 临时短气泡（≤7字）：常规事件提醒（骑手已取餐、预约确认中等），弹出后3-5秒自动隐藏，不可直接交互
- **FR-2.3**: 长气泡→自动收折：展开态≤30字/1行文字，收折态≤7-8字，收折后可点击，不再自动展开
- **FR-2.4**: 气泡频控：同一订单每个状态节点最多推送1次；待支付订单不推送气泡；用户可配置关闭
- **FR-2.5**: 气泡进入逻辑：点击气泡→解析order_id+event_type→加载订单卡片→基于event_type生成上下文引导

### FR-3: AI助手浮层界面
- **FR-3.1**: 浮层三种形态：9分屏（默认，顶部露出1/10底层页面）→全屏（上滑触发）→关闭（点击外部/关闭按钮/下滑到底部）
- **FR-3.2**: 顶部区域：AI Icon + "团小帮" + Slogan "全程陪伴，省心又省力" + 关闭按钮
- **FR-3.3**: 中间聊天区域：支持纯文本消息、订单卡片、功能卡片、系统消息滚动展示
- **FR-3.4**: 底部输入区：订单选择器标签、文本输入框、发送按钮（语音输入按钮占位但不实现功能）
- **FR-3.5**: 浮层交互手势：上滑展开全屏、下滑关闭、拖拽调整高度

### FR-4: 四层订单卡片架构
- **FR-4.1**: 基础层（必选）：缩略图+商品名称+价格+标签+距离/门店名+[导航][电话]+状态角标
- **FR-4.2**: 扩展层（可选）：按行业/状态动态展示（履约进度条、取餐码、配送地图、退款信息、酒店入住信息、景区入园信息、支付倒计时等）
- **FR-4.3**: 操作层（可选）：按行业×商品类型×状态状态机动态展示操作按钮（双模式：功能卡片模式/直接跳转模式）
- **FR-4.4**: 引导问题层（AI 独有）：最多3个预测问题标签，基于履约节点+多源反馈数据+对话上下文动态生成
- **FR-4.5**: 扩展层智能裁剪：首次展示完整；追问时仅展示被追问模块；气泡带入突出对应模块；多卡片隐藏扩展层；退款对话突出退款信息
- **FR-4.6**: 多卡片展示：精简卡片仅基础层+状态角标+主操作按钮，点击展开为完整四层卡片

### FR-5: 操作层全状态规则
- **FR-5.1**: 餐饮-待支付：操作层仅[立即支付]（红色强调按钮）
- **FR-5.2**: 餐饮-待使用-纯券码：[查看券码] + [订单使用提醒] + [提前预约免排队]
- **FR-5.3**: 餐饮-待使用-含点单：[立即点单] + [查看券码]（如有券码）+ [订单使用提醒]（含点单/配送时隐藏"提前预约"）
- **FR-5.4**: 餐饮-待使用-含配送：[立即配送] + [查看券码]（如有券码）+ [订单使用提醒]
- **FR-5.5**: 餐饮-配送中：扩展层配送进度+操作层[联系骑手][帮我加急]
- **FR-5.6**: 餐饮-已完成：[再来一单][去评价]
- **FR-5.7**: 餐饮-退款中：RefundingInfo + [帮我加急]（催退款）
- **FR-5.8**: 酒店预售券-待预约：标签行（含早/无早+整单未约随时退）+ [立即预约]
- **FR-5.9**: 酒店日历房-预订确认中：HotelStayInfo + [帮我加急]
- **FR-5.10**: 酒店日历房-预订成功：HotelStayInfo + [入住指引]
- **FR-5.11**: 景区团购-待使用：[查看券码] + [订单使用提醒] + [一站式游玩攻略]
- **FR-5.12**: 景区日历票-预订成功：景区名称行 + [一站式游玩攻略]
- **FR-5.13**: 综合-待使用：[查看券码] + [订单使用提醒] + [提前预约免排队]
- **FR-5.14**: 旅行社-预约成功：出行信息行 + [出行指引]
- **FR-5.15**: 操作按钮双模式：功能卡片模式（设置提醒/预约/加急等需确认的操作展开聊天内卡片）vs 直接跳转模式（查看券码/联系骑手/导航等简单操作）
- **FR-5.16**: 退款按钮不外露：操作层和引导问题中均不展示"申请退款"，仅通过自然语言通道触发

### FR-6: 退款能力与挽留机制
- **FR-6.1**: 退款入口仅自然语言触发：用户说"申请退款""我要退款""退掉吧"等
- **FR-6.2**: AI 挽留干预：用户表达退款意图后，先展示挽留信息（优惠稀缺+库存稀缺+使用价值+退款损耗+替代方案），不直接触发退款
- **FR-6.3**: 挽留后用户决策：用户表示继续使用/改期→触发对应流程；用户再次明确"退掉吧"→不再二次挽留，触发退款流程
- **FR-6.4**: 退款能力判断：f(当前时间, 商家退款规则, 用户权益) → full_refund/partial_refund/non_refundable/cancelable
- **FR-6.5**: 退款申请卡片：展示退款金额+退款说明+原因选择（不想要了/商家原因/计划有变/其他）+[取消][确认退款]
- **FR-6.6**: 退款文案动态展示：退款提示文案由后端接口返回，前端不硬编码退款规则
- **FR-6.7**: 各场景挽留策略差异化：团购券待使用（优惠+有效期+使用引导）、预售券待预约（优惠+随时退+预约引导）、预售券已预约（使用价值+改期替代）

### FR-7: 待支付状态专项设计
- **FR-7.1**: 待支付订单详情页左下角不展示 AI 入口（展示应付金额），不推送气泡
- **FR-7.2**: 待支付仅可通过订单列表右上角入口进入 AI 助手
- **FR-7.3**: 待支付卡片：基础层+扩展层（支付倒计时内联：左侧倒计时+右侧红色[立即支付]按钮）+操作层仅[立即支付]
- **FR-7.4**: 待支付引导问题以消除顾虑为核心：P0「随时退吗？」、P1「商品包含什么？」、P2「门店在哪？」（最多3个）
- **FR-7.5**: 待支付取消挽留：用户表达取消意图时AI基于优惠稀缺+库存稀缺+活动稀缺进行挽留
- **FR-7.6**: 引导问题以消除支付顾虑为核心，支付通过卡片[立即支付]按钮完成，引导问题不承担促支付职能

### FR-8: 引导问题层
- **FR-8.1**: 每次最多展示3个引导问题，按反馈量降序排列
- **FR-8.2**: 数据驱动排序：基于CPO/CCR/IM/电话多源反馈数据的历史反馈量排序
- **FR-8.3**: 去重规则：用户已提问或AI已回答的问题不再重复展示
- **FR-8.4**: 动态更新：对话中根据用户意图动态调整，P3上下文追问可替换P2
- **FR-8.5**: 行业感知：同一履约节点不同行业展示不同问题
- **FR-8.6**: 正向牵引：高反馈量但不适合主动引导的问题（投诉等）仅在用户主动提问时响应
- **FR-8.7**: 排除清单："用户未描述明确诉求"等无法转化的问题不展示
- **FR-8.8**: 排序算法：FinalScore = α×P0_score(离线反馈量0.5) + β×F_score(在线CTR0.3) + γ×C_score(上下文0.2)
- **FR-8.9**: 对话轮次联动：≤5轮展示P0+P1+P2+P3追问；>5轮展示1-2个；>20轮不展示引导问题
- **FR-8.10**: 全行业引导问题矩阵：覆盖餐饮/酒店/景区/综合/旅行社的各履约节点

### FR-9: 功能卡片
- **FR-9.1**: 使用提醒设置卡片：快捷选项（明天/后天/本周五六日/下周五六日，按当前日期动态计算）+自定义提前天数步进器（1-30天，默认3天）+提醒方式+[确认设置]+预览"将于X月X日10:00提醒您"；已设置状态展示[修改提醒][取消提醒]
- **FR-9.2**: 预约表单卡片：日期选择器（灰色标记已约满）+时段单选+人数步进器+联系电话+⚠️提示+[确认预约]；预约确认中→成功/失败异步更新
- **FR-9.3**: 加急请求卡片：自动生成加急文案+备注输入（≤50字）+[发送加急]；15分钟冷却时间；已发送状态展示发送时间
- **FR-9.4**: 再来一单确认卡片：商品信息+数量调整+优惠券选择+[立即下单]→跳转支付
- **FR-9.5**: 退款申请卡片：见FR-6.5
- **FR-9.6**: 攻略/指引卡片：AI动态生成游玩攻略/入住指引/出行指引内容，含导航/详情跳转按钮
- **FR-9.7**: 功能卡片通用规则：在聊天流中展开而非跳转页面；提交后AI回复结果消息；支持[取消]返回

### FR-10: 通用组件
- **FR-10.1**: FulfillmentTimeline履约进度条：N个节点N-1段，done(实心✓)/current(脉冲动画)/pending(空心)三种状态
- **FR-10.2**: CollapsibleTimeline可折叠进度：默认折叠，点击展开，用于已取餐/已送达等终态，背景#F3F4F6，圆角8px
- **FR-10.3**: RefundingInfo退款申请中：单行左侧状态文案+右侧退款金额
- **FR-10.4**: RefundSuccessInfo退款成功：单行左侧"已原路退回"+右侧退款金额，灰色文字
- **FR-10.5**: PickupCodeDisplay取餐码：有码时展示二维码+大字取餐码；无码时展示"查看取餐码"按钮
- **FR-10.6**: 支付倒计时内联：左侧倒计时+右侧红色立即支付按钮，嵌入扩展层
- **FR-10.7**: 配送地图卡片（AI独有）：骑手实时位置+距离标注+实时刷新

### FR-11: 多轮对话上下文管理
- **FR-11.1**: 上下文结构：session_id + current_order_id + order_context(行业/商品类型/状态/扩展层/操作按钮) + message_history + resolved_questions + pending_action
- **FR-11.2**: 会话级过期：关闭浮层后30分钟内重新打开恢复上下文；超过30分钟新建会话
- **FR-11.3**: 跨订单切换：message_history保留；resolved_questions清空；pending_action中断；引导问题重新匹配；退款上下文清空
- **FR-11.4**: 上下文与引导问题联动：新会话P0+P1+P2；对话中已回答移除+P3补充；>5轮1-2个；>20轮不展示

### FR-12: 气泡推送与卡片状态同步
- **FR-12.1**: WebSocket实时推送：用户在AI助手前台+订单状态变更时，服务端推送order_status_update事件
- **FR-12.2**: 15s轮询兜底：WebSocket不可用时降级为轮询
- **FR-12.3**: 卡片状态更新规则：配送中→已送达（扩展层切换CollapsibleTimeline+操作层变更+引导问题更新）
- **FR-12.4**: 时序保证：气泡携带latest_status与卡片状态不一致时，以卡片WebSocket实时状态为准
- **FR-12.5**: 边界处理：用户操作功能卡片时不打断，后台更新+操作完提示；退款对话中不中断对话流程

### FR-13: AI降级策略
- **FR-13.1**: L1部分降级（超时>3s/错误率>5%）：引导问题降级为静态规则；上下文降级为单轮问答；功能卡片/操作按钮正常
- **FR-13.2**: L2完全降级（服务不可用/错误率>20%）：隐藏引导问题层（四层→三层）；禁用自然语言输入（输入框提示"AI助手升级中，请使用下方按钮操作"）；功能卡片禁用，全部直接跳转；退款禁用自然语言→提示去详情页；气泡正常推送
- **FR-13.3**: L3引导问题降级：仅引导问题排序降级为静态规则，其他正常
- **FR-13.4**: 降级恢复：健康检查通过后灰度恢复10%→30%→50%→100%，每5分钟一级；不中断当前会话

### FR-14: 异常处理与边界条件
- **FR-14.1**: 订单数据加载失败→卡片区域展示"订单信息加载失败"+[重试]
- **FR-14.2**: AI意图解析失败→展示订单选择器引导手动选择
- **FR-14.3**: 门店名为空→不展示门店行；距离为空→不展示距离；标签超3个→只展示前3个
- **FR-14.4**: 点单+配送并存→按优先级展示（配送>点单）
- **FR-14.5**: 退款状态与履约状态并存→退款状态优先
- **FR-14.6**: 无法获取子状态→降级为基础状态展示

### FR-15: UI/UX规范
- **FR-15.1**: 卡片圆角12px，白色背景
- **FR-15.2**: 操作按钮白色背景#FFFFFF + 灰色边框1px solid #E5E7EB + 灰色文字#6B7280 + 圆角8px
- **FR-15.3**: "立即支付"使用红色背景+白色文字（唯一红色按钮）
- **FR-15.4**: 引导功能按钮垂直堆叠、右对齐、间距6px，样式同主操作按钮
- **FR-15.5**: 状态角标颜色：橙色=待支付/待确认，红色=待使用/待核销，蓝色=进行中，绿色=已完成，灰色=已取消
- **FR-15.6**: 导航图标为SVG斜箭头(-45°)，14×14px，#6B7280
- **FR-15.7**: 所有操作按钮统一白色背景，不使用蓝色渐变背景（仅"立即支付"红色）
- **FR-15.8**: AI助手品牌头像使用现有AIAssistantIcon SVG组件

### FR-16: 后端服务
- **FR-16.1**: 独立后端项目，技术栈Node.js + Fastify + TypeScript + SQLite + Prisma ORM
- **FR-16.2**: REST API：订单CRUD、AI对话接口、引导问题获取、退款能力判断、提醒设置、预约提交、加急请求、再来一单、对话历史
- **FR-16.3**: WebSocket：订单状态实时推送、气泡推送通知
- **FR-16.4**: 数据持久化：订单数据、对话历史、提醒设置、预约记录使用SQLite存储
- **FR-16.5**: 引导问题排序算法实现（离线权重+CTR反馈+上下文调整）
- **FR-16.6**: 退款能力判断接口（基于时间+规则+权益的mock判断）
- **FR-16.7**: AI对话mock引擎：基于规则匹配+意图识别+预置话术生成回复
- **FR-16.8**: 接口响应时间<500ms

## Non-Functional Requirements
- **NFR-1**: 前端首屏渲染时间 < 1s（本地开发环境）
- **NFR-2**: API 接口响应时间 < 500ms
- **NFR-3**: TypeScript 严格模式，零编译错误
- **NFR-4**: 浮层打开/关闭/形态切换动画流畅（300ms CSS transition）
- **NFR-5**: 代码与核心业务代码隔离（通过Git分支+独立组件目录），方便回退
- **NFR-6**: 移动端适配（375px 视口为基准，支持320-430px宽度）
- **NFR-7**: 前端组件化架构：AI助手相关组件独立放在 `src/components/AiAssistant/` 目录下
- **NFR-8**: 前后端类型共享或对齐

## Constraints
- **Technical**: 前端必须复用React 18+TypeScript+Vite技术栈；后端使用Node.js+Fastify+TypeScript+SQLite+Prisma；不引入额外重型UI框架，继续使用纯CSS
- **Business**: 必须复用已有AI助手入口，不得新建并行入口；"申请退款"按钮不得在操作层/引导问题外显；待支付订单不得在详情页展示AI入口和气泡
- **Dependencies**: 后端项目独立分仓管理；前端通过API调用与后端通信
- **Brand**: 助手名称"团小帮"，Slogan"全程陪伴，省心又省力"；参考V1.1设计风格但代码重写

## Assumptions
- 前端Mock对话阶段使用预置规则话术，不接入真实大模型
- 后端数据使用预置的mock数据（基于现有mock.ts的16种场景扩展）
- WebSocket在Demo阶段使用mock状态变更模拟推送
- 极端天气正向引导用mock数据展示
- 游玩攻略/入住指引/出行指引等AI生成内容使用预置模板
- 所有跳转操作（导航、电话、支付页面）在Demo中使用toast/弹窗模拟
- 二维码使用qrcode库（已有依赖）生成
