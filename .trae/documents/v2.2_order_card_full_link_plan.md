# 团小帮 v2.2 订单卡片全链路开发计划

## 一、项目调研结论

### 1.1 当前项目状态
- **技术栈**：React 18 + TypeScript + Vite
- **项目版本**：v2.1.0
- **现有核心模块**：
  - `AiOrderCardDemo.tsx`：订单卡片 Demo，含 30+ 订单示例、四维筛选器（行业×商品类型×履约方式×订单状态）、完整四层架构（基础层+扩展层+操作层+引导问题层）
  - `AiAssistantOverlay.tsx`：AI 助手浮层，支持 9 分屏/全屏拖拽、聊天消息流
  - `AiAssistantProvider.tsx`：全局状态管理（上下文、消息、订单、气泡等）
  - `OrderCenter.tsx`：订单中心页面（订单列表 + 订单详情）
  - `mock.ts`：订单 mock 数据（ORDER_LIST，现有 ~20+ 订单）
  - 功能卡片组件：`RedeemReminderCard.tsx`（使用提醒）、`ReservationInfoCard.tsx`（预约）、`VoucherCodeSheet.tsx`（券码）

### 1.2 核心差距分析
| 模块 | 现状 | 目标 | 差距 |
|---|---|---|---|
| **带入场景** | 有 openAssistant 接口，但未实现"订单详情页进入时默认发送订单卡片到聊天记录"功能 | 从订单详情左下角 AI 入口进入时，自动将当前订单卡片作为第一条消息发送到聊天流 | 需新增消息中的订单卡片渲染、自动发送逻辑 |
| **功能卡片** | 有 RedeemReminder/Reservation/VoucherCode 三个独立组件，未集成到聊天消息流中 | 7 类功能卡片（使用提醒、预约表单、加急请求、再来一单、退款申请、攻略/指引、通用规则）在聊天流中展开交互 | 需新增卡片类型、集成到消息流、补全新增卡片组件 |
| **完整卡片** | Demo 中有完整四层架构实现，但仅在 Demo 页展示，未集成到 AI 助手聊天流 | AI 助手聊天消息中可渲染完整四层架构的订单卡片 | 需将 Demo 中完整卡片逻辑抽取为可复用组件，集成到聊天消息 |
| **精简卡片** | Demo 中有 compact 模式，但未实际使用 | 订单列表/气泡入口等场景使用精简卡片 | 需完善精简卡片样式和交互 |
| **测试数据** | ORDER_LIST 约 20+ 订单，行业/状态覆盖不完整 | 按 行业×商品类型×履约方式×订单状态 四维矩阵构造无重复的完整测试订单集 | 需扩充 mock 数据，确保全矩阵覆盖 |

### 1.3 数据结构现状
- Demo 使用 `DemoOrder` 接口（AiOrderCardDemo.tsx），字段丰富，含扩展层类型定义
- 主项目使用 `OrderData` 接口（types.ts），字段较少，与 Demo 接口不统一
- **关键问题**：两套数据结构需统一，建议以 Demo 的 DemoOrder 为基础扩展为标准订单卡片数据模型

## 三、分模块开发计划

### 模块 1：带入场景（Entry & Bring-in）

**目标**：从订单详情页左下角 AI 入口进入时，自动将当前订单卡片发送到聊天记录中。

#### 3.1.1 功能说明
- 入口来源：`order_detail`（订单详情页左下角 AI 图标）
- 进入 AI 助手后，自动发送一条 assistant 消息，内容为当前订单的完整订单卡片
- 订单信息与订单详情页完全一致：商品名、价格、状态、适用门店、缩略图等

#### 3.1.2 涉及文件
| 文件 | 修改类型 | 说明 |
|---|---|---|
| `src/components/AiAssistant/AiAssistantProvider.tsx` | 修改 | openAssistant 中新增 order_detail 来源的自动发送订单卡片逻辑 |
| `src/components/AiAssistant/types.ts` | 修改 | ChatMessage 新增 orderCard 字段，支持消息中渲染订单卡片 |
| `src/components/AiAssistant/AiAssistantOverlay.tsx` | 修改 | 消息渲染中新增订单卡片消息类型的渲染逻辑 |
| `src/components/AiAssistant/OrderCardMessage.tsx` | 新增 | 聊天消息中的订单卡片组件（从 Demo 抽取） |

#### 3.1.3 实现步骤
1. 统一订单卡片数据模型：将 `DemoOrder` 接口合并/升级到标准 `OrderData`（或新建 `OrderCardData`）
2. 新增 `OrderCardMessage` 组件：从 `AiOrderCardDemo.tsx` 的 `renderOrderCard` 抽取为独立可复用组件
3. 修改 `ChatMessage` 类型：新增 `orderCard?: OrderCardData` 字段
4. 修改 `AiAssistantProvider.openAssistant`：当 `source === 'order_detail'` 且有 `orderId` 时，自动插入一条带订单卡片的 assistant 消息
5. 修改 `AiAssistantOverlay` 消息渲染：支持 `orderCard` 类型消息的渲染
6. 确保订单数据一致性：从 `fetchOrderById` 获取的订单数据与订单详情页展示完全一致

#### 3.1.4 验收标准
- [ ] 从订单详情页点击左下角 AI 图标进入，自动打开 9 分屏浮层
- [ ] 聊天记录中第一条消息为当前订单的完整卡片（四层架构）
- [ ] 订单卡片信息与订单详情页完全一致（商品名、价格、状态、门店、缩略图等）
- [ ] 从订单列表页进入不自动发送订单卡片（保持现有行为）
- [ ] 关闭后重新进入同订单，不重复发送（基于 session 去重）

### 模块 3：完整卡片（Full Order Card）

**目标**：将 Demo 中的完整四层架构订单卡片集成到 AI 助手聊天流中，样式和功能与 Demo 完全一致。

#### 3.3.1 四层架构回顾
| 层级 | 组成 |
|---|---|
| **基础层** | 缩略图 + 商品名 + 价格 + 标签 + 门店行（导航/电话/距离） + 状态角标 |
| **扩展层** | 9 种类型：progress / hotel_stay / refund / payment_countdown / travel_info / pickup_code / delivery_completed / scenic_entry / refund_success |
| **操作层** | primary 主按钮 + secondary 次按钮 + 催用理由（预售券待预约专用） |
| **引导问题层** | 最多 3 个预测问题，胶囊形 chip |

#### 3.3.2 涉及文件
| 文件 | 修改类型 | 说明 |
|---|---|---|
| `src/components/AiAssistant/OrderCard/` | 新增目录 | 订单卡片组件统一目录 |
| `src/components/AiAssistant/OrderCard/FullOrderCard.tsx` | 新增 | 完整四层架构订单卡片（从 Demo 抽取） |
| `src/components/AiAssistant/OrderCard/OrderCardBase.tsx` | 新增 | 基础层组件 |
| `src/components/AiAssistant/OrderCard/OrderCardExtension.tsx` | 新增 | 扩展层组件（9 种类型渲染器） |
| `src/components/AiAssistant/OrderCard/OrderCardActions.tsx` | 新增 | 操作层组件 |
| `src/components/AiAssistant/OrderCard/OrderCardSuggestions.tsx` | 新增 | 引导问题层组件 |
| `src/components/AiAssistant/orderCardTypes.ts` | 新增 | 订单卡片类型定义（统一数据模型） |
| `src/components/AiAssistant/AiOrderCardDemo.tsx` | 重构 | 改为引用新的 FullOrderCard 组件，保持 Demo 功能不变 |

#### 3.3.3 实现步骤
1. 抽取 `orderCardTypes.ts`：定义统一的订单卡片数据模型（基于 DemoOrder 优化）
2. 抽取 `OrderCardBase`：基础层（缩略图、商品名、价格、标签、门店行、状态角标）
3. 抽取 `OrderCardExtension`：扩展层 9 种类型的渲染器
4. 抽取 `OrderCardActions`：操作层（primary/secondary 按钮 + 催用理由）
5. 抽取 `OrderCardSuggestions`：引导问题层
6. 组合为 `FullOrderCard` 组件
7. 重构 `AiOrderCardDemo.tsx`：改为使用新组件，确保 Demo 功能不变（验证抽取正确性）
8. 集成到 `OrderCardMessage` 中（模块 1 的输出）

#### 3.3.4 验收标准
- [ ] 完整卡片四层架构与 Demo 完全一致（像素级还原）
- [ ] 9 种扩展层类型均可正确渲染
- [ ] 门店行显示规则正确（有 hotel_stay/scenic_entry/travel_info 时隐藏）
- [ ] 催用理由在预售券待预约状态正确显示
- [ ] scenic_entry 景区专属双行样式正确（非 info-list）
- [ ] 按钮样式正确（30px 高度、15px 圆角、primary/secondary 区分）
- [ ] 状态标签红色文字、无背景色块
- [ ] 旅行社门店行无导航、无距离
- [ ] AiOrderCardDemo 原有功能完全正常（回归验证）

## 六、潜在风险与应对

| 风险 | 影响 | 概率 | 应对方案 |
