# v2.2 订单卡片样式与功能还原修复计划

## 背景

AI 助手聊天流中的订单卡片（FullOrderCard 组件）与 Demo 模式（AiOrderCardDemo）存在样式和功能差异。需要严格按照 Demo 设计还原，确保在所有行业、商品类型、履约方式、订单状态组合下，订单卡片的展示和交互与 Demo 完全一致。

---

## 一、已确认的差异点清单

### 1. 数据层：订单信息缺失（使用规则标签等）

**问题**：`convertOrderListItemToCardData` 从 OrderListItem 转换时，标签信息不完整。
- OrderListItem 有 `productRules` 字段，包含 `refundRule` 和 `validDate`
- 当前只取了这两个字段作为标签，缺少更多使用规则信息
- Demo 中的订单有更完整的标签体系

**影响文件**：`src/components/AiAssistant/orderDataAdapter.ts`

**修复方案**：
- 完善 `buildListItemTags` 逻辑，根据不同行业和商品类型补充标签
- 餐饮：免预约、随时退、过期自动退
- 酒店：不约可退、限时取消
- 景区：随买随用、随时退
- 旅行社：含机票、含住宿、纯玩无购物

---

### 2. 门店名称文字大小

**问题**：需要确认门店名称的字号、颜色是否与 Demo 完全一致。

**对比结果**：
- Demo: `.demo-store-name` — `color: #4e5969`, `font-size` 继承 12px
- 新组件: `.oc-store-name` — `color: #4e5969`, `font-size` 继承 12px
- **结论**：样式一致，无需修改

---

### 3. 操作按钮样式、布局、文案

**问题**：操作按钮的样式、布局、文案需要与 Demo 完全一致。

**对比结果**：
- 按钮高度 30px、圆角 15px、字号 12px、字重 500 — 一致
- primary 渐变色 `#FF8A8A → #FFA898` — 一致
- secondary 背景 `#f2f3f5`，文字 `#4e5969` — 一致
- 按钮顺序：secondary → urgeReason → primary — 一致
- 按钮图标（⏰/🎫 → SVG）— 一致
- **结论**：样式一致，无需修改

**文案问题**：
- 不同状态下的按钮文案需要通过数据适配器确保与 Demo 一致
- 当前 `buildListItemActions` 的文案基本正确，但需要与 Demo 的 `getActionButtons` 完全对齐

### 6. 状态颜色差异

**问题**：Demo 中所有状态颜色都是红色 `#ff6b6b`，但新组件按状态分别使用了不同颜色。

**Demo 代码**（orderCardDemo.css L262-283）：
```css
.demo-card-status { color: #ff6b6b; }
.demo-card-status.status-green { color: #ff6b6b; }
.demo-card-status.status-orange { color: #ff6b6b; }
.demo-card-status.status-blue { color: #ff6b6b; }
.demo-card-status.status-gray { color: #ff6b6b; }
```

**新组件代码**（orderCard.css L75-93）：
```css
.oc-card-status.status-green { color: #00B42A; }
.oc-card-status.status-orange { color: #FF7D00; }
.oc-card-status.status-blue { color: #165DFF; }
.oc-card-status.status-gray { color: #86909C; }
.oc-card-status.status-red { color: #F53F3F; }
```

**结论**：
- Demo 的状态颜色全部是红色，这看起来是 Demo 的临时样式（可能还没按状态区分）
- 新组件按状态区分颜色更合理
- 但用户要求"严格按照 Demo 还原"，所以需要确认
- **修复方案**：先按 Demo 全部改为红色，等后续需求明确再区分

## 五、风险与注意事项

1. **样式隔离**：修改 `orderCard.css` 时确保 `oc-` 前缀不影响 Demo 的 `demo-` 样式
2. **组件拆分**：新组件是拆分的 4 层架构，Demo 是单文件实现，结构一致但实现方式不同，需确保组合后效果一致
3. **渐进式修复**：按步骤逐一修复和验证，避免一次性修改过多导致问题定位困难
4. **功能不回退**：修复样式时确保交互功能（按钮点击、引导问题）正常工作
