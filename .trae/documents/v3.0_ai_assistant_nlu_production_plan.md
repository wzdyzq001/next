# AI 助手自然语言交互生产功能开发方案 v3.0

## 一、项目背景与目标

### 1.1 背景
当前 AI 助手全景交互图（InteractionMap）已包含四大模块共 50+ 个场景的完整 Mock 数据和场景配置，但这些场景仅支持手动点击切换查看，无法通过自然语言输入触发真实的对话交互。

### 1.2 目标
在 AI 助手用户界面中实现真正的自然语言问题咨询功能，用户可以通过自然语言输入向 AI 助手提出问题，系统能够智能识别意图并返回相应的回复（文本、订单卡片、功能卡片、快捷回复等）。

### 1.3 范围
- **包含**：帮我约、订单使用提醒、取餐码查询、配送进度查询四大模块的 NLU 交互
- **不包含**：真实后端 API 对接、真实 NLP/ASR 语义理解、移动端深度适配优化

### 2.2 当前 sendMessage 流程
```
用户输入 → addUserMessage → chat() API 调用 → addAssistantMessage(reply + quickReplies)
```
当前 chat API 指向 `http://localhost:3001/api/chat`，返回简单文本回复，无法识别业务意图并返回结构化数据（订单卡片、功能卡片等）。

```
SceneModule → Scene → SceneStep
  - SceneStep 类型: text, order_card, reservation_card, reminder_card, 
                    extension_only, order_list, quick_replies
```
已有 50+ 个完整场景配置，包含所有 Mock 数据。

## 三、总体设计方案

### 3.1 核心思路
在前端实现一个**本地 NLU 引擎**，基于关键词匹配 + 规则引擎，将 InteractionMap 中的场景配置转化为可执行的对话流程。

### 3.2 架构设计

```
┌─────────────────────────────────────────────────┐
│              AiAssistantProvider                │
│  (sendMessage - 现有调用 API 的入口)            │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│            NLU Engine (新增)                    │
│  ┌─────────────┐  ┌──────────────────────────┐ │
│  │ 意图识别器   │→│ 对话状态管理器             │ │
│  └─────────────┘  └──────────────────────────┘ │
│  ┌─────────────┐  ┌──────────────────────────┐ │
│  │ 实体提取器   │→│ 回复生成器                 │ │
│  └─────────────┘  └──────────────────────────┘ │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│       场景配置 + Mock 数据 (复用 InteractionMap) │
└─────────────────────────────────────────────────┘
```

### 3.3 技术方案
- **NLU 引擎**：纯前端实现，基于关键词匹配 + 规则引擎
- **数据来源**：复用 InteractionMap.tsx 中的 Mock 订单数据和场景配置
- **集成方式**：通过替换/增强 `sendMessage` 中的 `chat()` 调用实现

## 四、详细功能设计

### 4.1 NLU 引擎模块

#### 4.1.1 意图识别
支持识别以下 4 大类意图：

| 意图类别 | 关键词 |
|---------|--------|
| **预约 (reservation)** | 预约、帮我约、立即预约、订座、订位、预定 |
| **提醒 (reminder)** | 提醒、设置提醒、帮我提醒、别忘了、到期提醒 |
| **取餐码 (pickup_code)** | 取餐码、取餐、取号、号码、凭号、取餐号 |
| **配送进度 (delivery)** | 配送、物流、骑手、外卖、送到哪了、什么时候到、配送进度、还有多久、怎么还没到 |

#### 4.1.2 对话状态管理
- 维护当前对话的意图、已收集实体、当前步骤
- 支持多轮对话状态追踪
- 支持随时取消当前流程
- 状态保存在 ConversationContext 中

#### 4.1.3 实体提取
从用户自然语言中提取：
- **日期实体**：明天、后天、周X、下周X、X月X日、过期前X天
- **时间实体**：几点、X点、XX:XX
- **人数实体**：X人、X位、几个人
- **数量实体**：数字、汉字数字

#### 4.1.4 回复生成器
根据意图 + 状态 + 订单上下文，生成回复消息：
- 纯文本回复
- 文本 + 快捷回复
- 订单卡片 (OrderCard)
- 预约卡片 (ReservationInfoCard)
- 提醒卡片 (RedeemReminderCard)
- 功能卡片 (FeatureCard)
- 订单列表 (CompactOrderCard list)

## 七、关键设计决策

### 7.1 NLU 引擎开关
- 通过 `constants.ts` 中的 `USE_LOCAL_NLU` 配置控制
- 默认开启本地 NLU，可切换为真实 API 调用
- 向后兼容，不破坏现有 API 调用逻辑

### 7.2 对话状态存储
- 对话状态存储在 `ConversationContext.dialogState` 中
- 包含：当前意图、已收集实体、当前步骤、订单上下文
- 随聊天历史一起持久化到 localStorage

### 7.3 Mock 数据来源
- 优先复用 InteractionMap.tsx 中已有的 Mock 订单数据
- 新增 nlu/scenarioData.ts 作为数据层，与 InteractionMap 解耦
- 数据结构完全兼容 OrderCardData 类型

### 7.4 回复生成策略
- 每轮对话可能产生 1~N 条消息（模拟多步消息）
- 通过 setTimeout 模拟 AI 思考时间（300-800ms）
- 支持延迟发送多条消息，模拟真实对话节奏

## 八、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 关键词匹配准确率不足 | 用户体验下降 | 扩展关键词库，支持模糊匹配，提供兜底回复 |
| 多轮对话状态混乱 | 对话流程出错 | 明确状态机定义，每个状态有清晰的转移条件 |
| Mock 数据不一致 | 展示异常 | 统一从 scenarioData.ts 导出，保持单一数据源 |
| 修改 AiAssistantProvider 引入回归 | 现有功能异常 | 最小化改动，仅修改 sendMessage 入口，通过开关控制 |
| 性能问题（大量数据） | 响应变慢 | 数据按需加载，关键词匹配采用高效算法 |
