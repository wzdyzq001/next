## 1. 产品概述

### 1.1 产品定位

团小帮 AI 智能助手是一款嵌入抖音生活服务订单中心的智能对话式助手，基于本地 NLU（自然语言理解）引擎，为用户提供预约、提醒、配送查询、取餐码查询等一站式订单服务，降低用户操作成本，提升订单使用体验。

### 1.2 核心价值

| 价值维度 | 具体描述 |
|---------|---------|
| **效率提升** | 自然语言交互替代多步表单操作，预约/提醒设置从 5+ 步简化为 2-3 轮对话 |
| **转化提升** | 通过 AI 触达（气泡引导、触达条）主动唤醒用户，提升预约率和券核销率 |
| **体验升级** | 对话式交互更贴近用户习惯，降低学习成本，提升用户满意度 |
| **运营提效** | 标准化 NLU 意图识别与实体抽取，减少人工客服介入，降低运营成本 |

### 1.3 目标用户

- 抖音生活服务已购券/订单用户
- 需要预约到店服务的用户（餐饮、酒店、景区等）
- 担心券过期、需要使用提醒的用户
- 关注外卖配送进度的用户

### 1.4 适用场景

| 场景 | 入口 | 核心诉求 |
|------|------|---------|
| 订单列表页 | AI 悬浮图标、触达条 | 查看订单状态、发起预约/提醒 |
| 订单详情页 | AI 入口按钮 | 针对特定订单的深度服务（预约、催单、退款等） |
| 气泡触达 | 订单卡片气泡 | 主动引导用户完成高价值操作（预约、设置提醒） |

## 3. 功能模块详细需求

### 3.1 AI 对话引擎（NLU）

#### 3.1.1 功能描述

本地 NLU 引擎负责对用户输入的自然语言进行意图识别、实体抽取，并基于对话状态机驱动多轮对话流程，最终生成结构化响应。

#### 3.1.2 核心组件

| 组件 | 职责 | 关键文件 |
|------|------|---------|
| 意图识别器 | 识别用户输入对应的意图类型 | `intentRecognizer.ts` |
| 实体抽取器 | 从用户输入中提取日期、时间、人数、手机号等实体 | `entityExtractor.ts` |
| NLU 引擎 | 对话状态管理、意图路由、响应生成 | `nluEngine.ts` |
| 预约处理器 | 预约意图的多轮对话状态机 | `handlers/reservationHandler.ts` |
| 提醒处理器 | 提醒意图的多轮对话状态机 | `handlers/reminderHandler.ts` |
| 配送处理器 | 配送进度查询处理 | `handlers/deliveryHandler.ts` |
| 取餐码处理器 | 取餐码查询处理 | `handlers/pickupCodeHandler.ts` |

#### 3.1.3 对话状态机设计

**通用状态流转：**

```
idle → 识别意图 → 进入对应意图状态机 → 完成/取消 → idle
```

**状态机核心字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `currentIntent` | `IntentType \| null` | 当前对话意图 |
| `entities` | `Record<string, string>` | 已收集的实体信息 |
| `currentStep` | `string` | 当前步骤标识 |
| `orderContext` | `Record<string, unknown>` | 订单上下文 |
| `data` | `Record<string, unknown>` | 扩展数据 |

#### 3.1.4 NLU 响应结构

```typescript
interface NluResponseMessage {
  role: 'assistant';
  contentType: 'text';
  content: string;                    // 文本回复内容
  quickReplies?: GuidedQuestion[];    // 快捷问题（旧版，已逐步被 actions 替代）
  actions?: MessageAction[];          // 快捷操作按钮（新版）
  orderCard?: OrderCardData;          // 订单卡片
  orderList?: OrderCardData[];        // 订单列表
  featureCard?: FeatureCardData;      // 功能卡片
  reservationInfo?: ReservationInfoCardData;  // 预约信息卡
  redeemReminder?: RedeemReminder;    // 兑换提醒
  delay?: number;                     // 消息延迟发送（毫秒）
}
```

#### 3.1.5 上下文管理

- **会话级上下文**：`sessionId` 标识会话，`conversationTurns` 记录对话轮次
- **已解决问题追踪**：`resolvedQuestions` 数组记录已回答的问题，避免重复
- **订单上下文**：支持带入当前订单信息，NLU 可基于订单状态调整回复策略
- **关键步骤保护**：在 `waiting_*_confirm` 等关键步骤，非取消类新意图不打断当前流程

### 3.2 帮预约功能

#### 3.2.1 功能描述

用户通过自然语言对话，AI 助手协助完成到店预约。支持日期选择、时段选择、人数选择、手机号填写、预约确认等全流程。

#### 3.2.2 适用行业/订单类型

| 行业 | 订单类型 | 是否支持 |
|------|---------|---------|
| 餐饮 | 团购券、自提、外卖 | ✅ 支持 |
| 酒店 | 日历房、团购券 | ✅ 支持 |
| 景区 | 门票、跟团游 | ✅ 支持 |
| 丽人/休闲 | 团购券 | ✅ 支持 |

#### 3.2.3 状态机流程（6 步）

```
idle
  ↓
selecting_order  （选择订单：如有上下文则跳过）
  ↓
validating_order  （校验订单是否可预约）
  ↓
collecting_info  （收集预约信息：日期、时间、人数）
  ↓
collecting_phone  （收集手机号：复用上次填写的手机号）
  ↓
confirming  （确认预约信息）
  ↓
completed  （预约完成）
```

#### 3.2.4 业务规则

| 规则项 | 规则内容 |
|--------|---------|
| **预约范围** | 支持未来 7 天内的预约（含今天），超出范围提示用户调整 |
| **营业时间** | 默认 09:00 - 22:30，非营业时间时段不可选 |
| **人数限制** | 1-10 人，超出范围提示用户调整 |
| **已过期时段** | 今天已过的时间不可选 |
| **约满时段** | 热门时段（18:00、19:00、19:30）模拟"已约满"，引导用户选择其他时段 |
| **手机号记忆** | 上次预约的手机号存入 localStorage（key: `ai_assistant_last_phone`），下次自动回填 |
| **动态容错** | 用户输入模糊时（如"下午""晚上""早点"），自动匹配最近可用时段并确认 |
| **引导文案** | 追问时包含已收集字段的上下文信息（如"已预约7月6日，请问几点到店？"） |

#### 3.2.5 实体抽取能力

| 实体类型 | 支持的表达形式 | 示例 |
|---------|---------------|------|
| **日期** | 今天/明天/后天/大后天、周X/本周X/下周X、X月X日、数字日期 | "明天"、"下周一"、"7月10号" |
| **时间** | 整点、半点、模糊时段（上午/下午/晚上/中午/早点） | "18:30"、"下午6点半"、"晚上" |
| **人数** | 纯数字、中文数字、带单位表述 | "3人"、"五位"、"5个" |
| **手机号** | 11位手机号 | "13812345678" |

#### 3.2.6 交互亮点

- **信息确认卡片**：预约确认时以卡片形式展示完整预约信息（门店、日期、时间、人数、手机号）
- **取消/改约**：已预约订单支持取消预约和重新预约
- **二次确认**：预约确认和取消均为关键操作，需用户二次确认
- **滚轮选择器**：日期和时间选择统一使用 Roller 滚轮组件，提升屏效和操作效率

### 3.3 订单使用提醒

#### 3.3.1 功能描述

用户为即将到期的券/订单设置使用提醒，到期前通过系统通知提醒用户使用，减少券过期浪费。

#### 3.3.2 适用订单

- 待使用状态的团购券
- 待使用状态的代金券
- 有明确有效期的日历类产品（酒店、景区等）

#### 3.3.3 状态机流程（8 步）

```
idle
  ↓
selecting_order  （选择订单：如有上下文则跳过）
  ↓
validating_order  （校验订单是否可设置提醒）
  ↓
checking_existing  （检查是否已有提醒，避免重复）
  ↓
collecting_datetime  （收集提醒日期和时间）
  ↓
confirming  （确认提醒信息）
  ↓
completed  （提醒设置完成）
```

#### 3.3.4 业务规则

| 规则项 | 规则内容 |
|--------|---------|
| **提醒日期范围** | 从今天到订单有效期最后一天 |
| **提醒时间限制** | 有效期当天的提醒时间不得晚于 18:00 |
| **过去时间校验** | 提醒日期/时间不能早于当前时间 |
| **重复提醒校验** | 同一订单已有提醒时，询问用户是修改还是取消 |
| **快捷日期选项** | 提供"明天"、"后天"、"过期前1天"、"过期前3天"等快捷选项（动态计算有效性） |
| **数据持久化** | 提醒数据存入 localStorage（key: `ai_fulfillment_redeem_reminders`） |

#### 3.3.5 订单有效性校验

`reminderOrderValidator.ts` 提供完善的订单校验逻辑：

| 校验项 | 校验规则 |
|--------|---------|
| 订单状态 | 必须为"待使用"状态 |
| 有效期 | 必须有明确的有效期，且尚未过期 |
| 已核销 | 已核销订单不可设置提醒 |
| 退款中 | 退款中订单不可设置提醒 |

#### 3.3.6 提醒数据结构

```typescript
interface RedeemReminder {
  id: string;           // 提醒唯一 ID
  orderId: string;      // 关联订单 ID
  productName: string;  // 商品名称
  remindAt: number;     // 提醒时间（时间戳）
  validDate: string;    // 订单有效期
  status: 'active' | 'cancelled' | 'triggered';
  createdAt: number;
}
```

### 3.7 猜你想问

#### 3.7.1 功能描述

在订单卡片底部提供"猜你想问"快捷问题按钮，用户点击后问题作为用户消息发送，AI 返回对应解答，降低用户输入成本。

#### 3.7.2 交互规则

| 交互项 | 规则 |
|--------|------|
| **展示数量** | 最多展示 5 个问题，超出可左右滑动 |
| **点击行为** | 点击后问题内容作为用户消息发送，AI 返回固定引导文案 + 快捷操作按钮 |
| **滑动交互** | 支持鼠标按住拖拽左右滑动，3px 阈值区分点击和拖拽 |
| **拖拽与点击互斥** | 拖拽距离超过 3px 时不触发点击，避免误操作 |
| **按钮样式** | 圆角胶囊按钮，带星号图标前缀，红色 #ff7875 |

#### 3.7.3 问题内容示例

- "可以带零食进去吗"
- "周末可以用吗"
- "怎么取餐"
- "可以退吗"
- "有效期到什么时候"

### 3.8 AI 触达模块

#### 3.8.1 功能描述

基于用户订单状态和行为，在合适的时机通过气泡/触达条等形式主动触达用户，引导用户完成高价值操作（预约、设置提醒等），提升订单转化。

#### 3.8.2 触达形式

| 触达形式 | 位置 | 适用场景 |
|---------|------|---------|
| **订单卡片触达条** | 订单卡片顶部 | 高优先级、高确定性的场景（如待使用券即将到期） |
| **悬浮气泡** | 订单列表页右下角 | 引导性触达（如新预约功能引导） |

#### 3.8.3 触达场景

| 场景 ID | 场景名称 | 触发条件 | 引导目标 |
|---------|---------|---------|---------|
| `guideReservation` | 预约引导 | 待使用券/订单 + 未预约 + 用户首次浏览 | 引导发起预约 |
| `reservationProgress` | 预约进度 | 预约进行中 + 用户进入订单详情 | 展示预约进度、引导改约/取消 |
| `usageReminder` | 使用提醒 | 券即将到期（7天内）+ 未设置提醒 | 引导设置使用提醒 |

#### 3.8.4 触达引擎

**核心能力：**

- **匹配引擎**：基于订单状态、用户行为、时间窗口等多维条件匹配触达策略
- **优先级控制**：多触达策略同时命中时，按优先级排序展示
- **频控策略**：支持按用户/按订单维度的曝光频次控制，避免打扰
- **动态内容**：触达文案支持动态模板，根据订单信息实时渲染

**触达配置结构：**

```typescript
interface ReachConfig {
  reachId: string;                 // 触达唯一标识
  pointType: 'order_card_bar' | 'detail_bubble';  // 触达点位
  displayMode: 'guide_clickable' | 'info_display'; // 展示模式
  priority: number;                // 优先级（数字越小越优先）
  icon: string;                    // 图标
  shortText: string;               // 短文案
  longText: string | DynamicContentFn;  // 长文案
  bubbleType?: 'long' | 'short';   // 气泡类型
  collapseStrategy?: 'auto_collapse' | 'scroll_collapse';  // 折叠策略
  guideMessage?: GuideMessageConfig;  // 点击后引导文案
  actions?: MessageAction[];       // 快捷操作
  match?: (ctx: ReachMatchContext) => boolean;  // 匹配函数
}
```

#### 3.8.5 气泡折叠策略

| 策略 | 说明 |
|------|------|
| `auto_collapse` | 展示 N 秒后自动折叠为短文案 |
| `scroll_collapse` | 用户滚动页面时折叠，停止滚动后展开 |

### 3.9 订单状态映射

#### 3.9.1 功能背景

餐饮订单存在多种履约方式（自提、外卖、券核销），每种方式下的子状态不同，但用户需要一个统一的主状态来快速理解订单情况。订单状态映射模块将底层细分状态映射为用户可理解的主状态。

#### 3.9.2 主状态定义（餐饮）

| 主状态 | 文案 | 颜色 |
|--------|------|------|
| `pending_pay` | 待支付 | 橙色 |
| `unused` | 待使用 | 橙色 |
| `cancelled` | 订单取消 | 灰色 |
| `redeemed` | 交易完成 | 绿色 |
| `refunding` | 退款申请中 | 蓝色 |
| `refund_success` | 退款成功 | 灰色 |
| `refund_fail` | 退款失败 | 橙色 |

#### 3.9.3 子状态定义

**自提（self_order）：**
- `self_01_pending_accept` - 待商家接单
- `self_02_accepted` - 商家已接单
- `self_03_preparing` - 制作中
- `self_04_waiting_pickup` - 待取餐
- `self_05_picked_up` - 已取餐

**外卖（delivery）：**
- `delivery_01_pending_accept` - 待商家接单
- `delivery_02_accepted` - 商家已接单
- `delivery_03_preparing` - 商家备餐中
- `delivery_04_waiting_rider` - 待骑手取餐
- `delivery_05_delivering` - 配送中
- `delivery_06_delivered` - 已送达

**券核销（voucher）：**
- `voucher_redeemed` - 已核销

### 4.1 AI 助手面板

#### 4.1.1 面板模式

| 模式 | 说明 |
|------|------|
| `minimized` | 最小化（悬浮图标） |

#### 4.1.2 对话消息样式

| 消息类型 | 气泡方向 | 背景色 |
|---------|---------|--------|
| 用户消息 | 右对齐 | 蓝色渐变 |
| AI 消息 | 左对齐 | 白色 |
| 系统通知 | 居中 | 灰色文字，无气泡 |

### 4.2 UI 组件规范

#### 4.2.1 Toast 提示

- 定位：屏幕正中心 `position: fixed`
- 层级：`z-index: 9999`
- 时长：2 秒自动消失
- 样式：半透明黑色背景 + 白色文字

#### 4.2.2 按钮规范

| 按钮类型 | 样式 | 适用场景 |
|---------|------|---------|
| 主按钮（Primary） | 红色渐变背景 + 白色文字 | 主要操作（确认、提交） |
| 次按钮（Secondary） | 浅灰背景 + 深灰文字 | 次要操作（取消、更多） |
| 快捷操作（Action） | 白色背景 + 灰色边框 + 圆角胶囊 | AI 消息内的快捷操作按钮 |
| 引导主按钮（Guide Primary） | 白色背景 + 灰色边框 + 11px 字号 | 引导类操作（预约/提醒） |

### 4.3 拖拽交互规范

- **点击与拖拽区分**：3px 移动阈值，小于阈值视为点击，大于阈值视为拖拽
- **拖拽触发**：必须按住鼠标（mousedown）后移动才触发拖拽
- **拖拽状态**：拖拽时光标变为 `grabbing`，内容禁止选中
- **拖拽结束**：mouseup 或 mouseleave 时停止拖拽
- **点击防抖**：拖拽结束后异步重置拖拽状态，避免误触发点击

### 4.4 关键操作二次确认

以下操作需要用户二次确认：

- 预约确认
- 预约取消
- 提醒设置确认
- 提醒取消
- 退款申请

## 7. 异常处理与边界校验

### 7.1 NLU 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 无法识别意图 | 返回兜底回复，引导使用已知功能（预约/提醒） |
| 实体识别失败 | 追问用户，提供快捷选项 |
| 日期超范围 | 明确告知用户可预约范围，并提供建议日期 |
| 时段不可用 | 说明不可用原因（非营业/已过/约满），建议其他时段 |
| 订单不可预约 | 说明原因，引导使用其他功能 |

### 7.2 表单校验

| 校验项 | 校验规则 | 错误提示 |
|--------|---------|---------|
| 预约日期 | 7 天内 + 非过去 | "预约日期需在 7 天内，请调整日期" |
| 预约时间 | 营业时间内 + 非过去 + 非约满 | "该时段不可选，请选择其他时段" |
| 人数 | 1-10 人 | "人数需在 1-10 人之间" |
| 手机号 | 11 位有效手机号 | "请输入正确的手机号" |
| 提醒日期 | 有效期内 + 非过去 | "提醒日期不能晚于订单有效期" |
| 提醒时间 | 有效期当天 ≤ 18:00 + 非过去 | "提醒时间不能晚于有效期当天18:00" |

### 7.3 边界场景

| 场景 | 处理方式 |
|------|---------|
| 用户连续输入无效信息 | 3 轮后提供快捷选项，降低输入成本 |
| 网络异常 | 降级为纯本地 NLU，不依赖后端服务 |
| 页面刷新 | 从 localStorage 恢复预约手机号和提醒数据 |
| 多订单场景 | 提供订单选择器，用户选择后再进入对应流程 |
